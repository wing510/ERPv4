## 0. 一句話總結（最重要）

**手動開「寄賣案件」，多次出貨累加進同一案；結算產生應收；收回時用加工廠 Lot 認批，依案件分配政策拆到出貨行；送出前預覽確認；左側寄賣頁改為案件版，不再使用舊追蹤畫面。**

---

## 1. 目的（要解決的痛點）

- **一張出貨一筆追蹤** 不符合實務：同一店家、同一寄賣案常 **多次出貨、多張銷售單**。
- 店家 **不會照銷售單退貨**；倉管收貨認 **加工廠 Lot（標籤批號）**，不是 ERP 內部批號。
- **同一加工廠 Lot** 可能 **分多次出貨、不同單價**；收回時需有 **分配規則**（可因客戶／案件而異）。
- 退貨 **不只未售**：還有破損、效期、錯貨、合約終止等（**實物拉回公司倉**）。  
- **已售退貨**（客人退給店、貨留店裡）**不走收回**，改 **作廢結算／AR 折讓**（見 §9.3.1）。
- 收回時 **退回哪個倉庫** 需可指定（預設回原出貨倉，可改）。

## 4. 核心流程

```text
【手動開寄賣案】
    │
    ├─ 出貨（寄賣類銷售）→ 選案件 → 累加進「品項池」
    │
    ├─ 【結算】店家報賣出量 → 扣已結算 → 產生應收（每次結算 1 張 AR）
    │
    ├─ 【收回】依退回原因 → 扣未售或已結算 → 入庫（或後續報廢流程）
    │
    └─ 【結案】案件結束
```

**對外操作：** 選寄賣案、填 **加工廠 Lot + 數量**（＋退回原因、收回日期、退回倉庫等）。  
**對內系統：** 自動命中出貨行、系統批號、認列單價（畫面不讓店家選銷售單）。

## 6. 品項池（出貨累加）

每次 **寄賣出貨過帳** 且 **指定寄賣案** 時，每一出貨行寫入池子一筆：

| 內容 | 說明 |
|------|------|
| 品項、數量、單位 | |
| 系統批號 | 出貨扣庫用的 ERP 批號 |
| **加工廠 Lot** | 標籤批號（從批號主檔帶出） |
| 出貨單價、出貨日 | 供結算、收回認列 |
| 已結算量、已收回量 | 累計 |
| 關聯出貨／銷售 | 內部追溯用，操作畫面不暴露給店家 |

**批號主檔（Lots）** 新增 **加工廠 Lot** 欄位；加工入庫或 Lots 補登時填入。

**出貨頁：** 寄賣出貨須 **選擇要掛哪個寄賣案**（或先開案再出貨）。

---

## 7. 結算（沿用概念，改掛案件）

- 在 **寄賣案** 內操作，非舊追蹤單。
- **可多次結算**；一次 **多品項 + 數量**。
- 每次結算 → **1 張應收**（`amount_system` / `amount_due` 規則同現行財務模組）。
- 扣減品項池 **已結算量**；不可超過可結算量。

## 8. 分配政策（已定：依客戶／案件）

### 8.1 設定層級

1. **客戶主檔**：預設政策  
2. **寄賣案**：開案帶入，**可覆寫**  
3. **收回當下**：依 **該案** 上設定執行  

### 8.2 政策選項

| 選項 | 白話行為 |
|------|----------|
| **先出先退** | 同一加工廠 Lot、多筆出貨 → **出貨日早的未售先扣** |
| **高價先退** | 同一 LOT、不同單價 → **先扣單價較高的未售** |
| **有填單價才篩選** | 收回單有填單價 → 只在那個單價的池子裡，再依 **先出先退** |

### 8.3 未售類收回的匹配順序

1. **加工廠 Lot**（必填）  
2. 僅 **未售可退量 > 0** 的池子行  
3. 若有填 **單價** → 篩選該單價  
4. 依本案 **分配政策** 排序後命中  
5. 入庫至該行 **系統批號**，並記 **認列單價、出貨行**（內部）

> **商業說明：** 店家常希望退貨算在較高單價批次；「先出先退」偏供應方邏輯。以 **客戶／案件政策** 寫死，避免現場爭議。

### 8.4 實作現況（2026-06）

| 項目 | 狀態 |
|------|------|
| DB 欄位（客戶預設、案件 `alloc_policy`） | 已有 |
| 開案帶入客戶預設 | 後端可帶；**開案 UI 未完整** |
| 收回依政策拆批 | **未完整**；收回 UI 改為 **逐筆池子（`pool_item_id`）** 選取，現場多為「選哪批扣哪批」 |
| `filter_unit_price`（有填單價才篩選） | 參數可存；**分配邏輯未用** |

## 9. 收回單

### 9.1 表頭欄位

| 畫面欄位 | 必填 | 說明 |
|----------|------|------|
| 寄賣案 | 是 | |
| **退回原因** | 是 | 見下表 |
| **收回日期** | 是 | 實際收到日 |
| **退回倉庫** | 是 | 見第 10 節 |
| 備註 | 視情況 | 改倉庫時 **必填** |
| 單價 | 否 | 店家退貨明細有價格才填 |

### 9.2 明細

- **加工廠 Lot + 數量**（必要時加品項）

### 9.3 退回原因與系統行為

| 退回原因（收回頁） | 白話 | 扣減對象 | 庫存 |
|----------|------|----------|------|
| **未售收回** | 還沒賣掉拉回公司 | 未售池 | 可售入庫 |
| **合約終止** | 不做了，剩的全拉回 | 未售池 | 可售入庫 |
| **商品瑕疵** | 壞了不能用 | 未售池 | 入 **所選倉**（見第 10 節） |
| **效期不足** | 快到期或已過期 | 未售池 | 同上 |
| **出貨寄錯** | 品項錯、送多 | 未售池 | 依狀況 |
| **其他原因** | 上列未涵蓋 | 未售池 | 依狀況 |

- **加工廠 Lot + 分配政策**：用於上述 **未售池收回**（實物回公司倉）。  
- **第二版** 在同一收回框架補齊 **破損／效期／錯貨** 等原因（皆計入品項池 **已收回**）。

### 9.3.1 已售退貨（不走收回）

| 項目 | 定案 |
|------|------|
| 實務 | 客人退給**寄賣店**，貨多半**留在店裡**可再賣，**不拉回公司倉** |
| 系統 | **不作** Return 收回、**不入庫** |
| 財務 | **作廢結算**（AR 須未收款）或 **AR 折讓**（已收款） |
| 再賣出 | 店家再次報量 → 正常 **結算** |

### 9.4 收回預覽（已定案）

- 按 **「確認收回」** 前，後端會 **preview 驗證**（數量、改倉備註等）；失敗顯示具體原因，**不清空表單**。  
- 定案原意：確認視窗列出將扣哪次出貨、單價、系統批號、退回倉庫。  
- **現行**：收回下拉已改 **一筆池子一選項（`pool_item_id`）**，選項即認批依據；確認框仍為 **通用文案**（未列分配明細表）。多行同批加總超量由 **前端加總檢查** 擋下。  
- **不做**「每改一個字就重算」的即時預覽。

---

## 10. 退回倉庫（進階 — 已定案）

| 情境 | 規則 |
|------|------|
| **未售、合約終止等（實物回倉）** | 預設 **原出貨那批的倉**；可改到其他 **使用中** 的倉 |
| **破損、效期** | **必選** 退回倉庫；從 **倉庫主檔現有倉** 下拉（請自行建立「報廢倉」「待驗區」等名稱） |
| **改與原出貨倉不同** | **備註必填** |
| **誰能改倉** | **不限主管**（與寄賣結算／收回相同可操作角色即可） |

**暫不做：** 倉庫主檔的「報廢／待驗用途」標記；日後若要自動限制下拉選單，再另開需求。

**入庫口徑：** 仍掛 **原系統批號**；**退回倉庫** 決定庫存進哪一倉（跨倉時備註留底）。

## 11. 畫面規格（寄賣模組）

1. **案件列表** — 客戶、狀態、未結算／未售摘要  
2. **案件明細** — 品項池、出貨／結算／收回歷史（合併「歷史」）  
3. **結算** — 品項 + 數量 + 日期  
4. **收回** — 原因、日期、退回倉庫、LOT、數量、可選單價 → **確認視窗** → 過帳  
5. **開案** — 客戶、分配政策（帶客戶預設）、備註  

**移除：** 舊 **CT 追蹤** 列表與「品項現況 · CT-xxx」操作流程；**不保留**雙軌 UI。
