## 0. 一句話總結（最重要）

**出貨照舊扣庫；一般銷售與寄賣共用同一套「應收/收款」；寄賣先追蹤出貨貨流，實際售出再結算產生應收，收款可多次，能收不滿也能結案（有規範），逾期提醒門檻可在公司設定調。**

---

## 1. 目的（要解決的痛點）

- **一般銷售**：出貨後要能建立應收、分次收款、看未收/逾期、結案。
- **寄賣**：貨先出去、錢不一定回來；要追蹤：
  - 這批寄賣貨在哪（跟哪張出貨相關）
  - 已售出結算多少、已收多少、尚欠多少
  - 可多次結算、可多次收款
  - 可結案（包含收不滿的例外情境）
- **主管/行政**：儀表板可快速看「未結案寄賣」與「逾期未收」，優先追款。

---

## 2. 與現有 v4.1 的關係（不衝突保證）

- **不改動既有庫存核心**：出貨過帳仍寫入 `inventory_movement` 的 `SHIP_OUT`（扣庫）
- **保留 SO → SHIP 1:N 架構**：一張銷售單可多次出貨、部分出貨既有邏輯不改
- 新增功能以「新增表/新增頁面/新增 API」方式接上，避免破壞既有流程

---

## 3. 左邊選單位置（已定案）

- **銷售**
  - 銷售單（SO）
  - 出貨（Shipment）
  - **寄賣（Consignment）** ← v4.2.1 新增（放銷售底下子選單）
- **財務**
  - **應收帳款 / 收款（AR）** ← v4.2.1 新增

> 直覺：寄賣屬於銷售流程；收款/應收屬於財務。

---

## 4. 核心流程（全局流程圖）

```text
銷售單 SO（可多次出貨）
  │
  └─ 出貨 SHIP（過帳 → SHIP_OUT 扣庫）
        │
        ├─ 一般銷售（NORMAL）
        │     └─ 出貨過帳 → 自動建立應收 AR
        │             └─ 收款（可多次）
        │                     └─ 結案（收滿或例外結案）
        │
        └─ 寄賣（CONSIGNMENT）
              └─ 出貨過帳 → 建立寄賣追蹤（先不建 AR）
                      └─ 寄賣結算（可多次、一次多品項）
                              └─ 每次結算 → 產生 1 張 AR
                                      └─ 收款（可多次）
                                              └─ 結案（收滿或例外結案）

第二階段：寄賣未售退回
  └─ 寄賣退回單 → 未售數量減少 → 庫存加回 → 再結案
```

## 6. 一般銷售 vs 寄賣：共用「應收/收款」，來源不同（已定案）

### 6.1 一般銷售（NORMAL）

- **應收建立時機：出貨過帳就自動建立 AR**
- AR 可多次收款，收滿結案
- 逾期提醒依 AR/出貨日計算（門檻可調）

### 6.2 寄賣（CONSIGNMENT）

- 出貨過帳：建立 **寄賣追蹤**（不立刻建 AR）
- **可多次結算**
- **每次結算產生 1 張 AR**
- AR 可多次收款，收滿結案
- 逾期提醒建議以 AR/結算日計算（門檻可調）

---

## 7. 金額欄位設計（已定案：並存 `amount_system` + `amount_due`）

為同時滿足「自動不易錯」與「可處理議價/折讓/例外」，採並存：

- **`amount_system`（系統金額）**
  - 系統自動算
  - **不可手改**
  - 用來對照（出貨/結算明細加總結果）
- **`amount_due`（應收金額）**
  - 預設 = `amount_system`
  - 允許調整，作為實際追款基準

### 7.1 系統金額來源

- 一般銷售：`amount_system` = 出貨明細加總
- 寄賣：`amount_system` = 該次結算明細加總

### 7.2 調整護欄（必做）

- 只有指定角色可調整（見「權限」）
- **原因/備註必填**
- 留 **修改人、修改時間、改前/改後**
- `amount_due` **不可小於** `amount_received`（已收款）

---

## 8. 收款（可多次）與狀態（共用）

### 8.1 收款規則

- 一張 AR 可登記多筆收款（多次）
- `amount_received` = 收款明細加總
- 狀態建議：
  - OPEN：尚未收
  - PARTIAL：部分收款
  - SETTLED：已收滿或結案完成

### 8.2 逾期天數（用途與計算）

**用途**：在列表/儀表板標紅「拖太久沒收」優先追款。

- 逾期天數 = 今天 −（起算日 + 門檻 N 天）
- 大於 0 才算逾期，並標紅

**起算日建議：**

- 一般銷售：AR 建立日（通常=出貨過帳日）
- 寄賣：AR 建立日（通常=結算日）

### 8.3 逾期門檻可調（已定案）

放在 **公司設定 → 財務/收款提醒**：

- 一般銷售逾期門檻 N（預設建議 14）
- 寄賣逾期門檻 N（預設建議 30）
- **到期前提醒 N**（預設 5；0＝關閉）：距離逾期門檻尚餘 N 天內，AR 列表與 Dashboard 標示「即將逾期」

## 9. 寄賣追蹤與結算（已定案）

> ⚠️ **歷史章節（v4.2.1 第一版）**：以下 §9.1「一張出貨＝一筆 CT 追蹤」**已不再使用**。現行為 **v4.2.2 寄賣案件**（手動開案、出貨掛案、池子累加、結算／收回掛案件）。§9.2「多次結算、每次一張 AR」概念仍適用。

### 9.1 寄賣追蹤單位（v4.2.1 第一版 · 已取代）

- **以出貨單（POSTED SHIP）為追蹤單位**：一張出貨單建立一筆寄賣追蹤  
- **現行**：見 `ERPv4.2.2` §5–§7；舊 CT 追蹤 UI 已移除

### 9.2 寄賣結算粒度

- **一次結算可包含多品項**（同一張結算單有多筆明細）
- 同一寄賣追蹤可「多次結算」
- **每次結算 → 產生 1 張 AR**
- 結算可只結算部分品項/部分數量，剩下下次再結算

---

## 10. 結案規則：收不滿 100% 也能結案（已定案：兩條路並行）

### 路徑 A：調整應收金額後結案（優先建議）

- 將 `amount_due` 調整為實際應收
- 收到該金額 → 正常 SETTLED

### 路徑 B：強制結案 + 寫原因（例外）

- 允許強制結案（例如：呆帳、爭議款、老闆決定不追）
- **原因必填**、留歷程、差額可見
- 權限限制（見下一節）

---

## 11. 權限（已定案：會計 + ADMIN/CEO/總務都有權限）

### 11.1 可查看

- **會計 / ADMIN / CEO / 總務（GA）**：可看應收、收款、寄賣追蹤與結算

### 11.2 可操作（登記收款、調整應收、強制結案）

- **會計 / ADMIN / CEO / 總務（GA）**：都可做
- 仍需遵守護欄（原因必填、不可小於已收、留歷程）

### 11.3 寄賣操作

- 寄賣結算（新增結算、產生 AR）：同上四者皆可

## 12. 分兩階段開發（已定案）

> **實作現況（摘要）**：財務 AR／收款、寄賣 `CONSIGNMENT` 銷售類型、儀表板提醒等 **已完成**；寄賣貨流改 **v4.2.2 案件版**（非本章 CT 追蹤 UI）；促銷見 `ERPv4.2.3`。

### 第一階段（MVP）— 先把「追款」做起來

**必交付：**

- 銷售單：新增 `CONSIGNMENT`
- 銷售→寄賣頁：
  - 寄賣追蹤列表（依出貨）
  - 多次結算（一次多品項）
  - 每次結算產生 1 張 AR（可點到財務）
- 財務→應收/收款頁：
  - AR 列表/明細
  - 多次收款登記
  - `amount_system` + `amount_due` 並存與調整規則
  - 正常結案 + 強制結案
  - 逾期天數計算與標紅（門檻由公司設定控制）
- 儀表板：
  - 未結案寄賣 / 逾期未收提醒（可先簡版）

**刻意不做：**

- 寄賣未售退回「自動入庫」完整流程（放第二階段）
- 付款（AP）、對帳報表（未列入 MVP）

### 第二階段 — 未售退回與進階帳務

- 寄賣退回單：
  - 退回明細（品項/數量/lot）
  - 庫存加回（新增 movement 類型）
  - 寄賣追蹤未售數量同步更新
  - 必要時與結算/AR 的調整規則補齊
- 視需求加入：折讓/退貨更標準流程、財務報表
