## ERP v4.2.8 財務 · AR 批次收款 — 討論定案說明書

> **狀態**：**MVP 已實作**（2026-07-05）；§7.2 完整版部分待做（手動改分配、匯出）  
> **版本**：**v4.2.8**（財務 · AR 批次收款：同客戶、多張應收、一筆匯款）  
> **日期**：2026-07-05  
> **範圍**：同一客戶多張未結清 AR，以**一筆匯款總額**一次登記收款，系統自動分配到各 AR  
> **前置**：`docs/ERPv4.2.1_寄賣與財務收款_討論定案說明書.md`、`docs/ERPv4.2.2_寄賣案件_討論定案說明書.md`（每次結算 → 1 張 AR）；**v4.2.5** 月結回饋／折抵不改本版範圍  
> **關聯**：不改「一結算一 AR」；不改月結回饋／折抵在結算當下的規則

---

## 0. 一句話總結（最重要）

**客戶每月匯一筆總額時，會計在 AR 模組勾選同客戶多張未結清應收，填收款日與匯款總額，系統依起算日由舊到新自動分配，一次寫入多筆收款明細；總額可小於勾選應收尚欠合計（甲），不可大於合計。**

---

## 1. 目的（要解決的痛點）

| 現況 | 痛點 |
|------|------|
| 每次寄賣結算產生 **1 張 AR** | 同一客戶一個月可能 **多次結算** → 多張 AR |
| AR 模組 **一張 AR 登記一次收款** | 客戶匯款單常只寫 **總額**，不列各結算單號 |
| 會計實務 **每月約一次** 對帳入帳 | 逐張 AR 重複 key 收款日／備註，易漏、易對不齊 |

**本方案**：保留每張 AR 的追溯與作廢規則，只在 **收款操作** 增加「批次入帳」。

---

## 2. 與現有系統的關係

### 2.1 不變的部分

| 項目 | 定案 |
|------|------|
| 寄賣結算 | 仍 **每次結算 → 1 張 AR**（`AR-STL-{結算單號}`） |
| 同一案件 | 仍可 **多次部分結算** |
| 單張 AR | 仍可 **多次收款**（分期收同一張） |
| 月結回饋／折抵 | 仍在 **結算提交** 或 **月結回饋產生** 時調整應收；批次收款只處理 **已產生 AR 的尚欠** |
| 作廢結算 | 仍依現規：AR **尚未收款** 才可作廢結算 |
| 調整歷程 | 折讓、強制結案等仍走 **例外處理**；分配尚欠以 **調整後應收 − 已收** 為準 |

### 2.2 新增的部分

| 項目 | 定案 |
|------|------|
| 操作入口 | **AR 應收帳款／收款** 模組 |
| 能力 | 同客戶 **多選 AR** → **一筆匯款總額** → **自動分配** → 一次提交 |
| 底層 | 仍寫入 **`ar_payment`**（每 AR 一筆或多筆）；不另建「合併 AR」 |

```text
結算₁ → AR₁ ─┐
結算₂ → AR₂ ─┼→ 批次收款（一筆匯款總額）→ 分配 → 各 AR 收款明細
結算₃ → AR₃ ─┘
```

## 3. 業務假設（已定案）

| 假設 | 內容 |
|------|------|
| 頻率 | **每月**約一次匯款入帳 |
| 匯款單 | 通常只寫 **總額**，不列各結算／各 AR 單號 |
| 客戶 | 一次批次只處理 **同一客戶** 的 AR（不可跨客戶勾選） |
| 幣別 | 第一版：同客戶 AR 幣別一致；若不一致則不允許同批（實作時依主檔／AR 幣別欄位檢查） |

---

## 4. 操作流程（畫面／步驟）

### 4.1 篩選與勾選

1. 進入 **AR 應收帳款／收款**。
2. 篩選：**客戶**、**未結清**（未收＋部分收）；可選加 **起算日區間**（例如本月）方便對帳。
3. 勾選要納入本批的 AR（或「全選此客戶未結清」）。
4. 按 **批次收款**（按鈕名稱實作時可再定）。

### 4.2 填寫與預覽

| 欄位 | 必填 | 說明 |
|------|------|------|
| 收款日 | 是 | 銀行入帳日／登帳日 |
| 本次匯款總額 | 是 | 客戶實際匯入金額 |
| 備註 | 否 | 建議填匯款單號或「2026-07 月結匯款」等，會寫入各筆收款備註（可加共用前綴） |

**預覽區**（提交前必顯示）：

| 顯示項 | 說明 |
|--------|------|
| 勾選 AR 筆數 | 例如 3 筆 |
| 尚欠合計 | 各 AR「應收金額 − 已收」加總 |
| 本次匯款總額 | 使用者輸入 |
| 差額 | 尚欠合計 − 總額（正數＝尚有未分配／下次再收） |
| 分配明細表 | 每 AR：起算日、來源、改前尚欠、**本次分配**、改後尚欠、狀態 |

### 4.3 確認提交

- 使用者確認預覽無誤 → **提交**。
- 系統 **同一交易** 寫入多筆 `ar_payment`，並更新各 AR 的已收／狀態。
- 成功後：刷新列表；可選顯示批次摘要（實作細節見 §7）。

## 5. 分配規則（已定案）

### 5.1 排序

- 僅對 **已勾選** 且 **未結清** 的 AR。
- 排序：**起算日（`ar_date`）由舊到新**；同日再以 AR 單號穩定排序。

### 5.2 尚欠計算

每張 AR：

```text
尚欠 = max(0, 應收金額 amount_due − 已收 amount_received)
```

（已含折讓等調整後的 `amount_due`，與現行 AR 一致。）

### 5.3 總額與尚欠合計的關係（核心定案）

| 情況 | 是否允許 | 行為 |
|------|----------|------|
| **總額 = 尚欠合計** | ✅ | 依序填滿；所勾 AR 全部結清（若每張都填滿） |
| **總額 < 尚欠合計** | ✅ **（定案甲）** | 依序填滿能填的 AR；**最後一張可能只收一部分**；其餘維持部分收／未收，**下次再收** |
| **總額 > 尚欠合計** | ❌ | **不允許提交**；提示超出金額，請改總額或減少勾選 |
| **總額 = 0 或負數** | ❌ | 不允許 |

**定案甲理由**：符合「每月只匯一部分」或「先付舊單」的實務；不必一次收滿所有勾選 AR 才能入帳。

### 5.4 分配演算法（示意）

```text
剩餘 = 本次匯款總額
FOR 每張 AR（起算日由舊到新）:
  分配[AR] = min(AR.尚欠, 剩餘)
  剩餘 -= 分配[AR]
  IF 剩餘 <= 0: BREAK
```

提交後：`amount_received += 分配[AR]`；若 `amount_received >= amount_due` → 狀態 **SETTLED**，否則 **PARTIAL**。

## 6. 邊界與例外

| 情境 | 處理 |
|------|------|
| 勾選中含 **已結清** AR | 不允許勾選或提交前剔除 |
| **跨客戶** 勾選 | 不允許；批次僅限同一 `customer_id` |
| 某 AR 已有 **調整歷程** | 仍可批次收款；尚欠以現行 `amount_due` 為準 |
| 尾差（例如尚欠 1000.01，匯 1000） | 不阻擋批次（甲）；若需清零頭，對該 AR 另用 **例外處理 → 調整應收** 後再收 |
| 溢收（匯多） | 整批不過；避免無 AR 可掛的餘額 |
| 作廢結算 | 與現行一致：該結算 AR 若 **已有收款** 不可作廢結算，須先 AR 處理 |
| 一般銷售 AR（出貨產生） | 第一版可 **允許** 與寄賣 AR 同批（同客戶、同幣別）；若僅想收寄賣，由使用者勾選 |

## 8. 與其他模組的介面

| 模組 | 關係 |
|------|------|
| **寄賣結算** | 只產 AR；批次收款不改結算單 |
| **月結回饋** | 折讓／折抵在回饋或結算時已反映於 `amount_due`；批次收款不重算回饋 |
| **經銷次月折抵** | 結算時扣 AR；批次收款不再扣折抵餘額 |
| **Dashboard** | 未收／逾期仍以 AR 為單位；批次收款後各 AR 狀態更新即可 |
| **Help** | 實作後更新 `js/help.js` AR 小節 |

## 9. 成功判斷（UAT 建議）

1. 同客戶 3 張 AR，尚欠 1000／2000／3000；匯 **6000** → 三張皆結清。  
2. 同上；匯 **2500** → 第一張結清、第二張收 1500（部分收）、第三張未收。  
3. 同上；匯 **7000** → 提交失敗，提示超出。  
4. 分配順序：起算日較早的 AR 先被填滿。  
5. 提交後各 AR **收款紀錄**、**調整歷程**（若有）仍正確；列表 **已收／狀態** 正確。  
6. 其中一 AR 已有部分收款 → 尚欠正確，批次只補不足部分。
