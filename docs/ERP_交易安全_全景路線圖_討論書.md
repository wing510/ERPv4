# ERP 交易安全：全景路線圖（寄賣／月結／AR）— 討論書

> 日期：2026-07-08  
> 文件定位：**討論用共識文件**（Phase 切分、Gate 標準、範圍邊界）  
> 注意：本文件**不是規格書**；細節（欄位、流程、例外）以現行程式 `js/`、`server/src/`、`server/sql/` 為準。

---

## 1. 目標（要達成什麼）

把「庫存 → 出貨／收回／結算 → 應收（AR）→ 收款／折抵／沖銷」這條金流鏈，逐段收斂成：

- **單一 DB 交易（或等價 RPC）**：避免半套資料
- **可稽核**：每次折抵／沖銷有正式分錄（不可只改餘額）
- **可冪等**：重送／重試不會重複入帳
- **POST/VOID 對稱**：作廢能完整反向（含庫存、AR、累積、折抵）
- **操作相容**：使用者畫面操作與流程盡量不變（除非另開核准）

---

## 2. 非目標（這份路線圖不承諾什麼）

- 不承諾本次就把「一般 × 寄賣 × 促銷／經銷 × 月結 × AR」一次改完
- 不承諾一次切換 Node → RPC 取代所有舊流程
- 不承諾此文件內容可取代程式碼或成為唯一真相

---

## 3. 北星（全景圖）

```
庫存異動 ──► 出貨／收回／結算 ──► 應收(AR) ──► 收款／折抵／沖銷
                │                    │
                ├─ 促銷計價          ├─ Dealer Credit
                └─ 經銷累積／月結     └─ 月結回饋入帳
```

最終所有「關鍵寫庫路徑」都應可追溯、可回滾、可作廢，且在並行下不會死鎖或產生重複入帳。

---

## 4. 現況地圖（誰在哪）

| 區塊 | 使用者感受（現行） | 交易安全現況 | 下一目標 |
|------|-------------------|--------------|----------|
| 一般出貨 NORMAL | 過帳／作廢照舊 | SQL 草稿可測；Node 未切 | Phase 1：切 RPC（僅 NORMAL） |
| 寄賣 CONSIGNMENT | 案件／結算／收回照舊 | 出貨 RPC 不納入；寄賣另線 | Phase 2：pool 進出＋結算原子化 |
| 促銷 | 一般出貨＋寄賣結算都有 | 一般出貨計價可 DB 化；寄賣仍偏 Node | Phase 1/2 各自 parity |
| 經銷／累積 | 等級價、累加、月結回饋 | ledger 草稿＋出貨折抵草稿 | Phase 1 驗證；Phase 3 收斂 |
| 月結 | 月結回饋／統計作業 | 作廢護欄先行；作業本體未重寫 | Phase 3 |
| AR | 應收、收款、調整、結案 | 出貨建／關 AR 可 DB 化；收款仍 Node | Phase 4（AR Phase） |

## 5. 操作相容性 Gate（接 Node 時硬性規則）

1. **不擅自改** UI、按鈕、欄位、操作步驟、API request/response contract（除非另核准）
2. 新 RPC **只導 NORMAL 出貨**；CONSIGNMENT 維持現行流程
3. 促銷、經銷價、Dealer Credit、累積採購：必須先做 **parity 對照測試**
4. 發現口徑差：先列差異清單並對齊，不可自行更改商業規則
5. Node 必須把 SQL exception 轉成使用者看得懂的訊息，不能直接吐 DB 錯誤

## 6. 分階段路線圖（Phase 0～5）

### Phase 0：不動使用者（現在～開分支前）

- 草稿 SQL 不接 Node、不上正式庫
- Git tag／還原點
- migration preflight：檢查舊資料是否會撞 UNIQUE（尤其是 AR 調整紀錄）

**成功判斷**：正式日常操作完全不變。

---

### Phase 1：出貨交易安全（第一槍）

**範圍**：僅 **NORMAL** 過帳／作廢（把整條 POST/VOID 收成單一交易）

包含（因為掛在出貨身上）：
- 一般出貨促銷計價 parity（DB 權威）
- 累積制經銷價 parity（牌價×等級%）
- Dealer Credit：出貨折抵＋作廢正式沖銷
- 一般出貨累積 ledger（POST/VOID）
- 出貨建 AR／作廢關 AR（跟該出貨綁定）
- 月結護欄（該月已月結則擋作廢）

不包含：
- 寄賣 pool、寄賣結算、寄賣收回
- AR 收款體系改革（VOID 狀態欄位化等）

**成功判斷**：
- DEV 上 NORMAL 過帳／作廢原子通過
- 與現行邏輯金額一致（或差異清單可解釋且經核准）
- 寄賣路徑零影響

---

### Phase 2：寄賣對稱（真正「一般 × 寄賣」）

範圍：
- 寄賣出貨進 pool（以及作廢／移除）
- 寄賣收回／改倉補正
- 寄賣結算過帳／作廢原子化（含促銷／經銷價／折抵）
- 結算建 AR／作廢與 pool 數量一致

**成功判斷**：寄賣任一關鍵步失敗，不會留下半套 pool／AR。

---

### Phase 3：月結／經銷帳本收斂

範圍：
- 月結回饋過帳／作廢走 ledger，避免與一般出貨雙算
- 月結統計與累積等級升級／待生效月份與出貨價一致
- 作廢護欄：安全優先，必要時再細化（需證據）

**成功判斷**：同一客戶「出貨累積＋月結回饋」可對帳，作廢順序清楚。

---

### Phase 4：AR Phase（應收本體）

範圍：
- 收款／作廢收款：正式 status（不再靠 `system_remark` 含 `VOIDED|`）
- 批次收款、沖銷差額、強制結案／重開規則穩定
- 調整 log 全面用 `source_type/source_id`
- 權限與 actor：一律由 server session 注入

**成功判斷**：財務對帳以狀態欄為準；AR 全生命週期一致可查。

---

### Phase 5：加固與正式切換

- request／payload hash
- 並行壓測、死鎖回歸
- 正式庫 migration Gate
- 監控與稽核報表

**成功判斷**：另開「正式 cutover Gate」通過後才上線。

## 7. 依賴關係（避免搶跑）

```
Phase 1 NORMAL 出貨 RPC
    │
    ├──────────────► Phase 4 AR（可部分並行：收款 status）
    │
    ▼
Phase 2 寄賣交易對稱
    │
    ▼
Phase 3 月結／ledger 作業收斂
    │
    ▼
Phase 5 正式加固與 cutover
```

## 8. 測試清單（Phase 1 必跑）

- **POST/VOID symmetry**：庫存、SO shipped_qty、AR、ledger、Dealer Credit 都對稱
- **rollback**：任一驗證失敗不留下半套
- **concurrency**：並行重試、重送、重入不重複入帳（冪等鍵有效）
- **pricing parity**：促銷／經銷價／折抵／累積與現行一致
- **鎖序／死鎖**：POST 與 VOID 的 lot lock 順序一致（VOID 要補 distinct lot_id + ORDER BY）

## 9. 已知技術債（本文件允許暫緩，不納入 Phase 1 Gate）

- AR 有效收款判斷仍依 `system_remark`（留 AR Phase）
- `p_actor` 仍為參數（Node 必須從 session 注入）
- CONSIGNMENT 新 RPC 仍不做（只導 NORMAL）
- request hash 未做（後續補 payload mismatch 防護）
