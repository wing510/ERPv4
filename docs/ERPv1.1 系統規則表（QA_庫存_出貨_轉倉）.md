## 系統規則表（QA／庫存／出貨／轉倉）

本文件用來統一「哪些情境可以做哪些操作」，避免各模組提示語、按鈕可用性、與實際資料寫入規則不一致。

---

## 名詞與欄位定義

- **Lot（批次）**
  - **`lot.status`（QA 狀態）**：`PENDING`（待QA）／`APPROVED`（QA已放行）／`REJECTED`（QA已退回）
  - **`lot.inventory_status`（庫存狀態）**：以系統規則判斷可用性（實作上常見：`ACTIVE`／`CLOSED`／`VOID`）
  - **`lot.expiry_date`（有效期）**：到期後視為不可用（VOID）
- **可用量（Available Qty）**
  - **來源**：以 `inventory_movement` 彙總為準（後端 `list_inventory_movement_available_by_lot` 或等價策略）。
  - **原則**：彙總成功時，若某 Lot 不在 map 中，視為 **0**（避免「缺資料」被誤當有庫存）。
  - **例外**：當彙總失敗（movementLoadFailed=true）才可以用 `lot.qty` 作為「估算顯示」，且 UI 應明確提示「可能不準」。

---

## 核心判斷（共用）

### 1) 是否到期

- **`isExpired(lot)`**：`lot.expiry_date < 今天` → 視為已過期
- **後果**：過期 Lot 視為 **不可用**（即使可用量 > 0 也要當作不可出貨/不可用）

### 2) 庫存狀態（建議口徑）

| 狀態 | 建議定義（口徑） | 常見後果 |
|---|---|---|
| `ACTIVE` | 未過期 且 可用量 > 0 | 可被挑選用於出貨/投料/轉倉/扣庫 |
| `CLOSED` | 可用量 <= 0（或已被扣完） | 不應出現在可用 Lot 清單（或不可選） |
| `VOID` | 已過期 或 被作廢 | 不可出貨/不可用；需要保留追溯 |

> 註：實作上 `inventory_status` 可能與彙總結果不同步，因此「顯示」與「可用判斷」建議以彙總＋到期規則為準。

### 3) QA 狀態文字（統一顯示）

| QA code | 顯示文字 |
|---|---|
| `PENDING` | 待QA |
| `APPROVED` | QA已放行 |
| `REJECTED` | QA已退回 |

## 操作規則表

### A) 出貨（Shipment）

#### A-1. Lot 可被用於出貨（Eligible for Ship）

同時滿足：

1. **QA**：`lot.status = APPROVED`
2. **未過期**：`!isExpired(lot)`
3. **可用量**：`availableQty(lot) > 0`
4. **庫存狀態**：等價於 `ACTIVE`（可用量與到期判斷後）

#### A-2. FEFO 自動分配（建議口徑）

- **排序規則**：`expiry_date ASC`（先到期先出）
- **篩選規則**：只取符合 A-1 的 Lot
- **拆單規則**：當單一 Lot 不足量，允許拆成多筆明細（多 Lot 共同滿足需求量）

---

### B) 委外加工（Process / Outsource）

#### B-1. 投料（扣庫 / PROCESS_OUT）

投料用 Lot 必須滿足：

1. `availableQty(lot) > 0`
2. `!isExpired(lot)`
3. **建議**：`lot.status = APPROVED`（若實務允許待QA投料，需明確定義例外與風險）

> 目前系統採「Lot picker 只顯示可用量 > 0」來減少誤選；若要強化，建議加上 QA gate 與效期 gate。

#### B-2. 回收（入庫 / PROCESS_IN）

- 回收會產生/更新 Lot 與異動，並需能追溯回投料來源（關聯資訊）。

---

### C) 庫存異動（Movements）

此處分成兩種主要類型：**一般扣庫/調整** 與 **轉倉**。

#### C-1. 一般扣庫（手動扣庫/報廢/調整 OUT）

建議必須滿足：

1. `availableQty(lot) >= 扣庫量`
2. `!isExpired(lot)`
3. `lot.status = APPROVED`（避免未放行批次被任意扣庫）

#### C-2. 轉倉（Transfer Warehouse）

轉倉會產生：

- **OUT**：從來源 Lot 扣庫（同 Lot）
- **IN**：在目標倉建立「新 Lot」並入庫（保留來源追溯）

**QA gate（目前採用的規則）**

| 轉倉型態 | 允許的 QA 狀態 |
|---|---|
| **全量轉倉**（轉出量 = 可用量） | `PENDING` 或 `APPROVED` |
| **部分轉倉**（轉出量 < 可用量） | 必須 `APPROVED` |

**目的**：避免 `PENDING` 批次被拆散到多倉、多子批次後，QA 管控與稽核難度大幅上升。

#### C-3. 轉倉後 QA 同步（目前採用的規則）

- 來源 Lot 與所有「由其轉倉衍生」的新 Lot，其 QA 放行/退回應保持同步（避免同源批次出現多種 QA 狀態造成稽核矛盾）。
- UI 層面：允許從任一衍生 Lot 進行 QA 放行/退回，但實際更新會同步回寫來源與其他衍生 Lot。

## UI/提示語建議（用於對齊各模組）

### 提示分段（建議）

- **第一段：`…流程：…`**（下一步/操作引導）
- **第二段：`…狀態：…`**（系統結果/是否可用/是否唯讀）

### 顏色語意（建議）

| 類型 | 顏色 |
|---|---|
| 警示（未載入/載入中/處理中/未過帳/待回收/可能不一致） | 棕色 |
| 成功/完成（已放行/已過帳/已結案） | 綠色 |
| 錯誤/作廢/取消 | 紅色 |
| 一般資訊（新單、提示文字） | 灰色 |
