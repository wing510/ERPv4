# ERP 規則清單（核心交易流程｜v4.1）

> 目的：把跨模組的核心流程規則集中，避免「流程很多、規則散、改一邊漏一邊」。  
> 範圍：採購、進口、收貨、批次/QA、庫存異動、銷售、出貨、加工、追溯。  
> 原則：只寫「流程/規則/常見誤區」，不寫程式細節與欄位英文名。

**v4.1 環境**
- 寫入交易：**必須走 bundle / command**（Node API），不可繞過直接改交易表。
- 資料表：Supabase `public` schema（例如 `goods_receipt`、`lot`、`inventory_movement`）。
- 未遷移模組：畫面會顯示**黃色橫幅**「此模組尚未遷移」；v4.1 主流程已遷移，**Split / Merge 寫入仍待補**。
- 防重：關鍵過帳/作廢需避免連點與重複送出（前端 in-flight 鎖 + 後端 idempotency）。

---

### A. 核心共通原則（所有交易流程都適用）

- **資料不可逆意識**
  - 過帳/扣庫/入庫/作廢沖銷，都會影響對帳；操作前要有明確提示，操作後要可追溯（`logs` + `inventory_movement`）。

- **避免重複送出**
  - 連點、網路重送、多人同時操作，必須能避免「重複過帳/重複扣入庫」。
  - 若後端回「已處理/重複」，前端應引導重新載入單據，不要只顯示錯誤卻留舊畫面。

- **作廢口徑**
  - 作廢不是刪除：作廢要留下原因與時間，必要時以沖銷方式還原庫存/狀態。

- **錯誤訊息口徑（交易通用）**
  - **缺資料**：指出缺什麼、在哪補。
  - **權限不足**：指出要找管理者或主管處理。
  - **狀態不允許**：例如已過帳/已作廢不可再做，訊息要明確。
  - **API 連線失敗**：先確認 Node API 是否在跑（本機 `npm run dev`），再 Ctrl+F5 重登。

---

### B. 採購 / 進口（Purchase / Import）

- **流程**
  - 採購：建 PO + 明細 → 收貨入庫引用 PO。
  - 進口：建報單 + 明細 → 進口收貨引用報單。
- **規則**
  - 作廢採購單/報單走 **cancel bundle**，不可直接刪除明細破壞狀態。
  - 報單狀態依申報量與已收量推導（OPEN / PARTIAL / CLOSED）。
- **常見誤區**
  - 進口「收貨入庫」應在 **Receive** 模組操作，不在 Import 頁直接入庫。

---

### C. 收貨入庫（Receive｜`goods_receipt` / `import_receipt`）

- **流程**
  - 先選來源（採購 PO / 進口報單）→ 填收貨明細與倉別 → **產生批次/入庫**（bundle）→ 必要時作廢。
- **規則**
  - 收貨必須選倉庫；明細數量不得為負。
  - 已作廢資料不可再被計入彙總或阻擋操作。
  - 過帳/作廢只走 `post_*_bundle` / `cancel_*_bundle`。
- **常見誤區**
  - 收貨頁看不到來源時，先確認 PO/報單狀態是否 OPEN/PARTIAL，以及 API 是否正常載入。

---

### D. 批次與 QA（Lots｜表 `lot`）

- **流程**
  - 收貨/加工回收產生批次 → QA 判定（放行/退回）→ 放行後才可被出貨/扣庫（依規則）。
- **規則**
  - QA 狀態屬高風險：放行等同允許出貨/扣庫，需有明確提示與可追溯紀錄。
  - 批次資訊（效期/倉庫/可用量）需一致呈現，避免選錯批次。
- **常見誤區**
  - 「可用量」與「總量」的差異要在畫面上清楚，避免誤出貨。

---

### E. 庫存異動（Movements｜表 `inventory_movement`）

- **流程**
  - 選擇用途/給誰（必要時必填）→ 選批次 → 輸入數量 → 扣庫/轉倉/調整。
- **規則**
  - 手動扣庫/調整屬高風險：需要求理由清楚，並可被稽核查出。
  - 轉倉若涉及 QA/狀態限制，需明確提示「何時可轉全部、何時不可」。
  - 轉倉走 `post_transfer_bundle`。
- **常見誤區**
  - 未放行批次是否可扣/可轉，需保持一致規則與提示。

---

### F. 銷售 / 出貨（Sales / Shipping）

- **流程**
  - 建銷售單 → 建立出貨 → 配批 → **過帳扣庫** → 必要時取消/作廢（沖銷）。
- **規則**
  - 過帳會扣庫，屬高風險；需避免重複過帳與不一致狀態。
  - 出貨人員等關鍵欄位必填，過帳一併寫入以便追溯。
  - 過帳/作廢只走 `post_shipment_bundle` / `cancel_shipment_bundle`。
- **常見誤區**
  - 過帳後再修改明細，容易造成對帳混亂；需用「取消/作廢」流程處理，而不是直接改資料。

---

### G. 加工（Outsource｜`process_order` 等）

- **流程**
  - 投料扣庫 → 回收產生新批次 → 必要時撤回投料/作廢回收/取消整單。
- **規則**
  - 投料與回收都會影響庫存與追溯，必須可查、可回補、不可重複送出。
  - 作廢/撤回必須留下原因，並把庫存/狀態回到一致。
- **常見誤區**
  - 同一張加工單的重複提交或重複回收，最容易造成「多扣/多入庫」。

---

### H. 追溯（Trace）

- **流程**
  - 以批次或交易 TX 為中心，查來源（從哪裡來）與流向（去哪裡）。
- **規則**
  - 追溯資訊應以「同一件事的一條鏈」呈現（能一眼看懂），避免只剩片段需要人工拼接。
  - v4.1 後端：`trace_lot_bundle` / `trace_transaction_bundle`。
- **常見誤區**
  - 若部分模組資料因權限或載入失敗而缺失，畫面應清楚提示「缺什麼/為何缺」而不是默默少一段。

---

### I. 倉庫庫存視圖（Warehouse Stock）

- **用途**
  - 依倉庫/產品彙總可用庫存，輔助出貨配批與現場查詢。
- **規則**
  - 數字應與 `lot` + `inventory_movement` 一致；有疑慮時用 Trace 或 Supabase 查 `lot` / `inventory_movement`。

### K. 最小回歸路徑（v4.1 本機建議）

1. 開 PowerShell → `cd d:\Desktop\ERP\server; npm run dev`
2. 瀏覽器 `http://127.0.0.1:5501/index.html` → Ctrl+F5 → admin 登入
3. 建主檔（產品/倉庫）→ 採購或進口 → 收貨入庫 → 產生批次 → QA 放行
4. 銷售/出貨過帳 → Trace 查 Lot / TX →（選做）作廢 → Logs 查操作紀錄
